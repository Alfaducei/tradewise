# New App Bootstrap — Global UI Template (Official shadcn Patterns)

**Use for every new app.** Drop at repo root as `NEW_APP_BOOTSTRAP.md`. Aligned with official shadcn docs: OKLCH colors, `@theme inline`, derived radius scale, MCP server + skill, RHF + Zod forms, theme-provider dark mode.

---

## 1. Core Rules (non-negotiable)

1. **All UI from shadcn/ui.** No custom Card/Button/Badge/Table/Input/Dialog/etc. No other UI libraries.
2. **Theming via CSS variables in `globals.css` only.** Colors, fonts, radius all live there. Tokens exposed to Tailwind via `@theme inline`.
3. **Icons: lucide-react only.** `h-4 w-4` inline, `h-5 w-5` in buttons.
4. **Charts via shadcn `<ChartContainer>`.** Never import recharts directly in feature code.
5. **Colors are OKLCH** in token definitions, not hex or HSL.
6. **Forms: React Hook Form + Zod + `<Controller>` + `<Field>`.** No other form libraries.
7. **Dark mode via theme provider** with `.dark` class toggle. No manual `prefers-color-scheme` media queries.

## 3. `components.json` — Canonical Shape

Generated by `init`, but know what's in it. **Three fields cannot change after init** (marked with 🔒 — would require deleting and reinstalling components to alter):

```json
{
  "$schema": "https://ui.shadcn.com/schema.json",
  "style": "new-york",                          // 🔒 "default" is deprecated
  "rsc": true,                                  // Next.js only; adds "use client"
  "tsx": true,                                  // false = .jsx output
  "iconLibrary": "lucide",                      // lucide only per our rules
  "tailwind": {
    "config": "",                               // BLANK for Tailwind v4
    "css": "src/app/globals.css",
    "baseColor": "neutral",                     // 🔒 neutral|stone|zinc|mauve|olive|mist|taupe
    "cssVariables": true                        // 🔒 always true for us
  },
  "aliases": {
    "components": "@/components",
    "utils": "@/lib/utils",
    "ui": "@/components/ui",
    "lib": "@/lib",
    "hooks": "@/hooks"
  },
  "registries": {
    // optional — add private/custom registries here
    // "@tradewise": "https://registry.tradewise.com/{name}.json"
  }
}
```

**Implication:** if you want to change `style`, `baseColor`, or `cssVariables` later, you delete `components/ui/*` and rerun `add` for every component. Pick correctly the first time.

## 6. Dark Mode — Theme Provider Setup

Dark mode works by toggling the `.dark` class on `<html>`. You need a theme provider.

### Next.js (App Router)

```bash
npm install next-themes
```

```tsx title="app/layout.tsx"
import { ThemeProvider } from "next-themes";

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en" suppressHydrationWarning>
      <body>
        <ThemeProvider attribute="class" defaultTheme="system" enableSystem>
          {children}
        </ThemeProvider>
      </body>
    </html>
  );
}
```

### Vite (React)

```tsx title="src/components/theme-provider.tsx"
import { createContext, useContext, useEffect, useState } from "react";

type Theme = "dark" | "light" | "system";
const ThemeContext = createContext<{ theme: Theme; setTheme: (t: Theme) => void }>({
  theme: "system",
  setTheme: () => {},
});

export function ThemeProvider({ children, defaultTheme = "system" }: {
  children: React.ReactNode; defaultTheme?: Theme;
}) {
  const [theme, setTheme] = useState<Theme>(
    () => (localStorage.getItem("theme") as Theme) ?? defaultTheme
  );
  useEffect(() => {
    const root = document.documentElement;
    root.classList.remove("light", "dark");
    const resolved = theme === "system"
      ? (window.matchMedia("(prefers-color-scheme: dark)").matches ? "dark" : "light")
      : theme;
    root.classList.add(resolved);
    localStorage.setItem("theme", theme);
  }, [theme]);
  return <ThemeContext.Provider value={{ theme, setTheme }}>{children}</ThemeContext.Provider>;
}
export const useTheme = () => useContext(ThemeContext);
```

Wrap `<App />` in `main.tsx`: `<ThemeProvider><App /></ThemeProvider>`.

### Toggle button (all frameworks)

```tsx
import { Moon, Sun } from "lucide-react";
import { Button } from "@/components/ui/button";
import { useTheme } from "next-themes"; // or your local hook

export function ThemeToggle() {
  const { theme, setTheme } = useTheme();
  return (
    <Button variant="ghost" size="icon"
      onClick={() => setTheme(theme === "dark" ? "light" : "dark")}>
      <Sun className="h-5 w-5 rotate-0 scale-100 transition-all dark:-rotate-90 dark:scale-0" />
      <Moon className="absolute h-5 w-5 rotate-90 scale-0 transition-all dark:rotate-0 dark:scale-100" />
      <span className="sr-only">Toggle theme</span>
    </Button>
  );
}
```

## 7. Forms — RHF + Zod + Field + Controller (Canonical)

This is the rule for **every form**. No alternative patterns.

### Setup

```bash
npm install react-hook-form zod @hookform/resolvers
npx shadcn@latest add field input textarea select checkbox radio-group switch button
```

### Template (adapt per form)

```tsx
"use client";
import { useForm, Controller } from "react-hook-form";
import { zodResolver } from "@hookform/resolvers/zod";
import * as z from "zod";
import {
  Field, FieldLabel, FieldDescription, FieldError, FieldSet, FieldLegend, FieldGroup,
} from "@/components/ui/field";
import { Input } from "@/components/ui/input";
import { Button } from "@/components/ui/button";

const schema = z.object({
  title: z.string().min(5, "Title must be at least 5 characters."),
  email: z.string().email("Enter a valid email."),
});
type FormValues = z.infer<typeof schema>;

export function MyForm() {
  const form = useForm<FormValues>({
    resolver: zodResolver(schema),
    defaultValues: { title: "", email: "" },
    mode: "onBlur",
  });

  const onSubmit = (data: FormValues) => {
    console.log(data);
  };

  return (
    <form onSubmit={form.handleSubmit(onSubmit)} className="space-y-6">
      <Controller
        name="title"
        control={form.control}
        render={({ field, fieldState }) => (
          <Field data-invalid={fieldState.invalid}>
            <FieldLabel htmlFor={field.name}>Title</FieldLabel>
            <Input {...field} id={field.name} aria-invalid={fieldState.invalid} />
            <FieldDescription>A short, descriptive title.</FieldDescription>
            {fieldState.invalid && <FieldError errors={[fieldState.error]} />}
          </Field>
        )}
      />
      <Controller
        name="email"
        control={form.control}
        render={({ field, fieldState }) => (
          <Field data-invalid={fieldState.invalid}>
            <FieldLabel htmlFor={field.name}>Email</FieldLabel>
            <Input {...field} id={field.name} type="email" aria-invalid={fieldState.invalid} />
            {fieldState.invalid && <FieldError errors={[fieldState.error]} />}
          </Field>
        )}
      />
      <Button type="submit">Submit</Button>
    </form>
  );
}
```

### Rules of thumb

| Rule | Why |
|---|---|
| Always wrap inputs in `<Field data-invalid={fieldState.invalid}>` | Styling + a11y |
| Always set `aria-invalid={fieldState.invalid}` on the control | Screen readers |
| Groups of checkboxes/radios: wrap in `<FieldSet>` + `<FieldLegend>` | Semantic HTML |
| Arrays (emails, phone numbers): use `useFieldArray`, key each item by `field.id` | RHF contract |
| Validation mode: `onBlur` for simple, `onChange` for live feedback | UX |
| For Select/Checkbox/Switch: don't spread `field` — use `field.value` + `field.onChange` | Controlled components |

- All components from @/components/ui/* — no custom primitives

- React Hook Form + Zod + <Controller> + <Field> — no exceptions

## Tools available (Claude Code)
- shadcn skill: auto-loaded when components.json exists
- shadcn MCP server: use to search/browse/install

  - "add sidebar and breadcrumb components"

## Before generating UI
1. Use MCP: "search shadcn for <component type>"
2. If found: npx shadcn@latest add <n>
3. If not: compose from existing shadcn primitives
4. Last resort: hand-build using tokens (bg-primary, etc.) — never hardcode

## File conventions
- shadcn primitives: src/components/ui/
- Feature components: src/components/features/<feature>/

## 9. Adding Custom Tokens (Official 3-Step Pattern)

```css
/* 1. Define in :root */
:root {
  --info: oklch(0.65 0.15 240);
  --info-foreground: oklch(0.99 0 0);
}
/* 2. Define in .dark */
.dark {
  --info: oklch(0.55 0.15 240);
  --info-foreground: oklch(0.145 0 0);
}
/* 3. Expose via @theme inline */
@theme inline {
  --color-info: var(--info);
  --color-info-foreground: var(--info-foreground);
}
```

`bg-info` / `text-info-foreground` now work anywhere.

| Want | Change |

## 13. Monorepo (if ever needed)

```bash
npx shadcn@latest init --monorepo
```

Structure: `apps/web/` + `packages/ui/`. Each workspace has its own `components.json`. Run `add` from the app dir: `cd apps/web && npx shadcn@latest add button`.

Imports use `@workspace/ui/components/button`. Keep `style`, `iconLibrary`, `baseColor` identical in both `components.json` files.

## 14. Custom Registry (optional, for cross-project components)

If you build reusable components across Voltage, Tradewise, etc., publish them as a shadcn registry:

1. Define `registry.json` with your items
2. `npx shadcn@latest build` generates `public/r/*.json`
3. Host on any static URL
4. Add to consumer projects' `components.json`:

```json
{
  "registries": {
    "@dj": "https://registry.yourdomain.com/{name}.json"
  }
}
```

Then `npx shadcn@latest add @dj/trades-table` in any project.
